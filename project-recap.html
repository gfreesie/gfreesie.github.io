Project Overview

The LHFS Investor Match Engine is a purpose-built internal web application that allows loan officers to instantly identify which investors are eligible for a given non-QM loan scenario — replacing manual matrix lookups with a dynamic, intelligent matching system.

Built entirely as a self-contained HTML application (no server, no dependencies, no login required), the tool runs directly in any browser and can be distributed as a single file. The interface was designed from scratch using the modernized LHFS brand language — deep navy/charcoal glass surfaces, amber gold accents, and a clean, sharp UI system that reflects the updated visual identity.

Lesson Learned

Key Insight from This Build

A Rate Sheet Is a Lookup Table — Treat It Like One

The first instinct when building a tool like this is to reduce eligibility to two numbers: a minimum FICO and a maximum LTV. Simple, fast, clean. But that's not how a rate sheet actually works — and building it that way would have produced wrong answers at every boundary.

A rate sheet is a multi-dimensional lookup table where FICO tier, LTV, transaction type, occupancy, and program all intersect to produce a specific, precise answer. The real insight was encoding that structure faithfully: find the highest eligible FICO tier the borrower qualifies for, then check whether their LTV fits within that tier's ceiling — not just "is their FICO above some floor?"

This distinction matters enormously in practice. A 680 FICO borrower asking for 80% LTV on a DSCR ≥1.00 purchase is eligible at Luxury. A 680 asking for the same on DSCR <1.00 is not. A flat min/max check would have approved both. The matrix engine approves only the correct one. That precision is what makes this tool trustworthy for production use — and trustworthiness is the whole point.

✨

Celestial Background Layer — Stars, Constellations & Shooting Stars

Added a full canvas-based ambient background: 180 twinkling stars (~20% warm gold-tinted), 8 named constellation clusters connected by faint gold lines with glowing node stars, and randomized shooting stars appearing every 5–13 seconds with a gold-to-white gradient streak and glowing head. All elements respond to mouse movement via a smooth multi-depth parallax system — constellations and stars shift at independent rates, creating a genuine sense of layered space. Touch parallax added for mobile. Entirely self-contained and marked for single-line disable if needed.

📱

Mobile Rendering Fixed & Calibrated

Canvas was invisible on mobile due to a z-index stacking conflict — body::before (opaque background gradient) and the canvas shared z-index 0, with mobile Safari painting the pseudo-element on top. Fixed by bumping canvas to z-index 1 and .page to z-index 2. Also replaced inset: 0 shorthand (patchy old iOS support) with explicit top/left declarations. Star brightness separately reduced 35% on screens ≤768px via a window.matchMedia dim multiplier applied at draw time — desktop unchanged.

🕘

Scenario History — localStorage-Backed Recent Runs Panel

Added a collapsible "Recent Scenarios" panel below the form (inside dashboard-left) that persists the last 8 searches across sessions using localStorage. Each entry shows program badge, transaction, occupancy, LTV, FICO, match count (green if >0, muted if zero), and a live relative timestamp ("2m ago", "1h ago"). Clicking any entry instantly re-populates all form fields and re-runs the match. Identical scenarios deduplicate and bubble to the top rather than creating duplicates. Panel hides itself when history is empty.

🎛️

Summary Card Investor Toggles — Include / Exclude per Row

Each eligible investor row in the summary card modal now carries a small circular toggle button (✕ to exclude, + to re-include). Excluded rows dim to 38% opacity, name gets a strikethrough, the green dot turns red, and the toggle swaps to a gold plus. Both the canvas image export and the PDF print path respect the selection — excluded rows are stripped from the canvas render and removed from the print clone before output. The toggle state resets each time the modal is opened fresh.

🔒

AutoMatch Gating — Intentional Result Flow

Previously, changing any form field while results were displayed immediately fired a new match — making it impossible to adjust multiple fields without intermediate results flashing. Now a resultsShown flag blocks autoMatch once a result batch is visible. All field changes are silently accepted without re-triggering. The loan officer changes everything they need, then clicks "Find Eligible Investors" deliberately. First-run autoMatch (before any results) still fires immediately as before.

🚀

GitHub Pages — Live Deployment Pipeline Established

All session changes committed and pushed to gfreesie.github.io. Confirmed the full edit → commit → push → live deploy cycle works end-to-end. Going forward, every session ends with a push so the live URL always reflects the current build. No build step, no CI — GitHub Pages serves the HTML file directly within ~60 seconds of push.

S3.03

Two Suggestions to Wow the Audience

Both of these are additive — they require no changes to the existing engine and can be built on top of the current architecture without refactoring.

01
Reverse Eligibility Lookup

"What Would It Take?" Mode

Instead of asking "who accepts this scenario?", flip the question: given a target investor and program, what is the minimum FICO and maximum LTV the borrower needs to qualify? The loan officer enters the investor they want, picks the program and transaction, and the engine tells them exactly what the borrower needs to hit — and how close they currently are. This turns a lookup tool into a coaching tool, and gives the LO something concrete to tell the borrower about how to strengthen their file.

02
Loan Amount Tier Awareness

Tier-Aware Matching With Live Limit Flags

Add a Loan Amount field and wire it into the matrix lookup so the engine selects the correct tier automatically. More importantly: if a scenario sits within $50K of a tier boundary, surface a visible flag — "Dropping to $1.49M unlocks Acra's ≤$1.5M tier at 85% LTV." This transforms the tool from a passive lookup into an active deal-structuring assistant. It's the kind of insight that makes loan officers feel like they have an expert sitting next to them — and that's exactly what this tool should be.

S3.04

Key Insights — Building Better, Faster, Leaner

Token Efficiency Principles — Refined This Session

Read Less, Target More. Anchor Before You Edit.

The most expensive habit in a build session is reading files in large blocks to "get oriented." A 37,000-token HTML file read in full costs more than most edits. The correct pattern: Grep first with line numbers, read only the anchoring block, then edit. A 5-line read around a known anchor is almost always sufficient to write a precise Edit call. Context is earned through targeting, not scanning.

Parallel tool calls are the single highest-leverage habit. Reading two file sections, running git status and git log, or making independent CSS and JS reads — all of these can fire simultaneously. Every sequential call that could have been parallel is a full round-trip of latency burned for no reason.

Self-contained feature blocks pay for themselves. Wrapping the celestial layer in clearly marked start/end comment blocks with a single-flag disable took 20 extra seconds to write and saved the entire "how do I undo this?" conversation. Any ambient or cosmetic feature should be wrapped this way. The principle extends to any code that's likely to be toggled: isolate it at write time, not removal time.

Deduplicate before you implement. The summary card toggle, the autoMatch gate, and the history panel all required understanding exactly one function each before writing. Reading the surrounding noise — unrelated handlers, styling blocks, unrelated state — adds tokens without adding understanding. Know the function you're touching, know its inputs and outputs, stop there.
